What this pipeline does, why it exists, and how it turns academic PDFs into something AI can actually search.

Meet the pipeline

Imagine you have six dense academic research papers as PDF files. You want to ask an AI: "What statistical methods did these papers use?" — and get a precise, sourced answer.

The problem: AI models can't read an entire 15-page paper in one go. They need text fed to them in digestible pieces. This pipeline is the machine that creates those pieces.

✦ Aha! Moment

Think of this like a librarian who pre-reads every book and writes summary index cards — one card per paragraph. When you ask a question, the AI searches the cards first, then reads only the relevant pages. This is called RAG.

What "RAG" actually means

Without RAG, an AI only knows what it was trained on — not your specific papers. RAG has three steps:

1

Chunk — break documents into pieces

Split each PDF into small, overlapping text segments (~380 tokens each). That's what this pipeline does.

2

Embed — give each chunk a numeric fingerprint

An embedding model converts each chunk into a list of numbers that captures its meaning. These numbers are stored in a vector database.

3

Retrieve & Generate — answer the question

When a user asks a question, the most relevant chunks are fetched and handed to a LLM to produce a grounded answer.

Your role in the pipeline

This project handles step 1 only — and it handles it very well. The output (a JSONL file of chunks) is ready to be loaded directly into any vector database.

MODULE 02

Meet the Cast

Two files, two jobs — and a data structure that holds everything together.

Two files, two responsibilities

📋
run.py — The Front Door

The CLI entry point. Reads your flags (--input, --max-tokens...), builds a config, calls the chunker, prints results. Think of it as the dashboard — you never need to modify it unless you want new controls.

🧠
chunker.py — The Brain

The importable library with all the logic: PDF extraction, heading detection, sentence splitting, chunk assembly. It knows nothing about your terminal — it just takes a PDF path and yields Chunk objects. This is the file you'd study, extend, and import.

Key insight for AI steering

When you tell an AI coding agent "add a new metadata field to every chunk", specify chunker.py, the Chunk dataclass. When you say "add a --verbose flag", specify run.py, the argument parser. Knowing which file does what makes your instructions precise and unambiguous.

The Chunk: a product label for text

Every piece of text the pipeline produces is wrapped in a dataclass called Chunk. Think of it like a product label attached to every piece of text — it carries not just the text but everything you'd want to know about it: where it came from, which page, which section.

chunker.py — Chunk dataclass
@dataclass
class Chunk:
    chunk_id: str
    text: str
    source_file: str
    page_start: int
    page_end: int
    section: str
    chunk_type: str  # "text" | "table" | "figure"
    chunk_index: int
    token_count: int

    def to_dict(self) -> dict:
        return asdict(self)
Plain English
  1. @dataclass — "Python, please auto-build the boring setup code for this data container."
  2. chunk_id: str — A unique fingerprint (MD5 hash) for this chunk. Used to look it up later.
  3. text: str — The actual text content. Always starts with [Section: Name], [Table: ...], or [Figure: ...] depending on chunk type.
  4. source_file: str — The PDF filename this chunk came from.
  5. page_start / page_end: int — Which pages the text spans, so you can cite your sources.
  6. section: str — Which section heading this chunk belongs to (e.g. "Methods").
  7. chunk_type: str — What kind of content this chunk holds: "text" for prose, "table" for camelot-extracted table data, or "figure" for a figure caption. Lets you filter by content type during retrieval.
  8. chunk_index: int — The sequential position of this chunk within its source file.
  9. token_count: int — How many tokens the text contains (used to verify the hard max was respected).
  10. to_dict() — Converts this object to a plain Python dictionary, for writing to JSONL.

ChunkingConfig: the control panel

Three numbers control the entire chunking behavior. You set them once and they flow through every decision the pipeline makes:

🎯
target_tokens = 380

The soft target. The pipeline tries to fill each chunk to about this size — roughly one dense paragraph from an academic paper. Not a hard limit; the pipeline may stop at a sentence boundary just before hitting this.

🛑
max_tokens = 512

The hard ceiling. No chunk will ever exceed this. Chosen to match the maximum input size of the text-embedding-ada-002 model family. Think of it as a maximum box size.

🔗
overlap_tokens = 60

The context bridge. The last ~1 sentence of each chunk is repeated at the start of the next one, so a topic that spans a chunk boundary isn't lost. About 60 tokens ≈ one short sentence.

🧹
garble_threshold = 0.02

The figure caption quality gate. PDF text extraction sometimes merges words together without spaces ("thekinasecascade"). The space-to-length ratio of the extracted text is measured; any figure caption scoring below 0.02 is silently dropped rather than added as a garbled chunk.

Steering tip

To get bigger chunks (more context per query hit): --target-tokens 512 --max-tokens 768. For finer-grained search: --target-tokens 200 --max-tokens 300. The defaults are tuned for dense academic text — you may need to adjust for other document types.

MODULE 03

Taming the PDF Beast

Why PDFs are a nightmare for text extraction — and how the pipeline beats them.

PDFs aren't really documents

Here's a fact that surprises most people: a PDF doesn't store text as sentences. It stores individual characters as drawing instructions — "draw the letter 'T' at position (72, 680), font size 9.8pt, in Arial." There's no paragraph, no sentence, no "this is a heading" concept baked in.

The photocopy analogy

Imagine someone took a book, photocopied every page, and gave you the stack of images. You can see words, but if you try to select text — you'd just be highlighting pixels. A PDF is like a slightly smarter version of that: it has the character positions, but no higher-level structure. Structure must be inferred from position, size, and repetition.

📐
No semantic structure

No native concept of "paragraph" or "heading." You have to infer these from visual properties like font size, position, and capitalization.

📰
Multi-column layouts

Most academic journals use two columns. Without detecting this, text from column A and column B get interleaved, producing incoherent sentences.

🏷️
Running headers/footers

Every page typically repeats the journal name, volume, and page number. Without stripping these, they'd appear as garbage inside your chunks.

📊
Figures and captions

"Figure 2A shows..." text is visually present but semantically noise for a RAG pipeline focused on the paper's findings.

Two-column detection: reading the room

pdfplumber gives us each word's horizontal start position (x0). In a two-column paper, words cluster in two distinct zones. The algorithm looks for this pattern:

LEFT COLUMN
x0 ≈ 36–72pt (left zone)
RIGHT COLUMN
x0 ≈ 310–340pt (right zone)

If both zones each have ≥5% of words and the right zone has ≥15% total → two-column detected. Left is read first, then right.

chunker.py — _detect_two_column()
def _detect_two_column(words, page_width):
    if not words:
        return False
    bins = Counter(round(w["x0"] / 10) * 10
                   for w in words)
    total = sum(bins.values())
    right_total = sum(cnt for x, cnt
        in bins.items()
        if x > page_width * 0.40)
    if right_total < total * 0.15:
        return False
    left_peak  = max((cnt for x, cnt
        in bins.items()
        if x < page_width * 0.40),
        default=0)
    right_peak = max((cnt for x, cnt
        in bins.items()
        if x > page_width * 0.40),
        default=0)
    return (left_peak >= total * 0.05
            and right_peak >= total * 0.05)
Plain English
  1. if not words: If the page is empty, it's definitely not two-column.
  2. bins = Counter(round(x0/10)*10 ...) Round every word's left-edge position to the nearest 10 points and count how many words start there. We're building a histogram of where words begin horizontally.
  3. total = total word count on this page.
  4. right_total = how many words start in the right 60% of the page. If less than 15% of words are in the right zone, it's single-column — return False.
  5. left_peak = the most common starting position in the left zone (the left column margin).
  6. right_peak = same, but for the right zone (the right column margin).
  7. return True if both peaks each account for ≥5% of all words — confirming two distinct column margins exist.

Table extraction: why pdfplumber isn't enough

Academic tables look simple on screen but are notoriously hard to extract. pdfplumber's built-in table detector frequently misidentifies abstract blocks or figure areas as tables, and it produces hollow rows with empty cells when borders are missing.

The new approach: camelot lattice mode

Camelot runs a single pass over the entire PDF before page-by-page extraction begins, using lattice mode — it detects the actual ruled lines printed on the page and uses their intersections to identify cells. The result is a dict[page_number → [rows]] that is consulted when each page is processed.

chunker.py — camelot extraction & validation
def _extract_tables_camelot(pdf_path):
    if not _CAMELOT_AVAILABLE:
        return {}
    tables = _camelot.read_pdf(
        str(pdf_path),
        flavor="lattice",
        pages="all")
    result = {}
    for t in tables:
        rows = [list(row) for row
                in t.df.values.tolist()]
        result.setdefault(
            t.page, []).append(rows)
    return result

def _is_useful_table(rows):
        # Reject: fewer than 4 non-empty cells
        # Reject: single cell > 200 chars (paragraph mis-detect)
        # Reject: fewer than 2 rows or 2 columns
    if not rows or len(rows) < 2: return False
    col_count = max(len(r) for r in rows)
    if col_count < 2: return False
    non_empty = [c for row in rows
                 for c in row if c.strip()]
    if len(non_empty) < 4: return False
    if any(len(c) > 200 for c in non_empty):
        return False
    return True
Plain English
  1. flavor="lattice" — Use line-intersection detection, not text-position heuristics. Only works on tables that have visible printed borders — which is the common case in academic PDFs.
  2. pages="all" — Process every page in one library call, returning results keyed by page number. More efficient than calling per-page.
  3. t.df.values.tolist() — Camelot stores each table as a pandas DataFrame. We convert it to a plain list-of-lists of strings.
  4. _is_useful_table() — fewer than 2 rows/cols: A 1×1 region is not a table. Likely a text box or figure boundary that camelot mistook for a bordered cell.
  5. fewer than 4 non-empty cells: A nearly-blank extraction — camelot found the border lines but nothing meaningful was inside. Rejected.
  6. single cell > 200 characters: The classic "paragraph in one cell" false positive — pdfplumber sometimes treats an abstract or caption block as a single-cell table. This guard rejects it.
  7. return True — Passed all checks: this is a genuine, useful table. It will be emitted as a chunk_type="table" chunk.
Design trade-off: lattice-only

Lattice mode only captures tables with visible cell borders. Tables rendered with spacing alone (stream mode) are not extracted. This is a deliberate conservatism: the false-positive rate of stream mode on academic PDFs is high, and a missed table is less harmful than a garbled chunk passed to the embedding model.

MODULE 04

The Heading Detective

How the pipeline finds section boundaries using three overlapping signals — and why it has to be conservative.

Why headings are the skeleton

Without section detection, all text from a paper becomes one giant block under "Introduction." With it, each chunk gets labeled — [Section: Methods], [Section: Results]. That prefix dramatically improves retrieval precision because the embedding model encodes the section name as part of the chunk's meaning.

The locksmith analogy

Heading detection is like a locksmith with three keys. If any one key opens the lock, it's a heading. But the locks have many anti-tamper guards — so false positives (treating body text as headings) are aggressively prevented, because a false positive silently fragments a section into two separate context bubbles, destroying retrieval quality.

SIGNAL 1
📖
Name Recognition

Does the line match a known canonical section name? Abstract, Introduction, Methods, Results, Discussion, Conclusions... If yes — it's a heading. No font analysis needed.

SIGNAL 2
📏
Font Size Sleuthing

Is the font ≥1.15× the body size but ≤2× it? 3–8 words? Mostly alphabetic? No trailing comma/colon? These together indicate a sub-heading that Signal 1 didn't catch.

SIGNAL 3
📢
ALL-CAPS Shouting

Is every word uppercase, 3–6 words, each word ≥2 characters? Not a DNA primer sequence? Some journals capitalize section headings instead of using larger fonts.

Signal 2 up close: the font-size test

Signal 2 is the most nuanced — it needs to detect enlarged headings while rejecting paper titles (which are also enlarged, just more so).

chunker.py — _is_heading(), Signal 2 block
    # Signal 2: font-size enlarged but
    # not title-scale (titles are ~2.5x body).
    # Upper bound of 2.0x excludes 24pt paper
    # titles while keeping 11-14pt headings.
    if (
        body_font_size > 0
        and body_font_size * 1.15
            <= font_size
            <= body_font_size * 2.0
        and 3 <= word_count <= 8
        and len(alpha_words)
            >= word_count * 0.70
        and not s.endswith(
            ("-", ",", ":", ";"))
        and not _CONTINUATION_ENDS
            .search(s)
    ):
        return True
Plain English — every condition explained
  1. body_font_size > 0 — We have a valid font size estimate to compare against.
  2. body_font * 1.15 ≤ font_size ≤ body_font * 2.0 — The line is at least 15% bigger than body text (larger = louder = probably a heading). But capped at 2× to exclude paper titles, which are typically 2.4–2.5× body size. Without this upper bound, the first author line of a paper would be detected as a heading.
  3. 3 ≤ word_count ≤ 8 — Real section headings are a phrase, not a single word and not a paragraph. Rejects both "Abstract" (single word — handled by Signal 1) and long body sentences.
  4. ≥70% alphabetic words — Rejects lines that are mostly numbers, symbols, or accession codes (e.g., "NCBI GSE12345 table S3").
  5. doesn't end with - , : ; — A line ending mid-phrase is a continuation, not a heading.
  6. _CONTINUATION_ENDS.search — Doesn't end with prepositions like "of", "for", "in" — which would mean the "heading" is actually the first clause of a body sentence.
  7. return True — All conditions passed: this is a section heading.
✦ Aha! Moment

This is defensive programming. Each condition rules out a specific category of false positive that was discovered during testing. The comment "Upper bound of 2.0x excludes 24pt paper titles" tells you exactly what bug was found and how it was fixed. Real engineering leaves these breadcrumbs.

The false positive problem

Academic papers are full of lines that look like headings but aren't. The pipeline has explicit guards for each known class of impostor:

🚫

Author/affiliation lines

Pattern: "Jane Doe 1,2" or "Smith J*". Rejected via a regex that catches "CapitalizedName CapitalizedName Digit" sequences.

🚫

Journal banner lines

"BMC Genomics", "Open Access", "Research Article", "UC Davis" — all explicitly rejected by a dedicated regex. These often appear at ALL-CAPS or enlarged font size because journals style them prominently.

🚫

DNA/RNA primer sequences

Biology papers often print primer sequences in ALL-CAPS: "ACGTCTAGAACGT". Without a guard, every primer would fire Signal 3. The DNA pattern ^[ACGTNURYKM...]+$ rejects pure nucleotide strings.

🚫

Figure panel labels

Figure panels are often labeled "A B C D" in ALL-CAPS. Signal 3 requires each word to be ≥2 characters — so single-letter labels ("A", "B") can never trigger a false heading.

MODULE 05

The Slicing Engine

How text is cut into chunks that are exactly the right size — not too big, not too small, with just enough overlap.

The Goldilocks problem

Why does chunk size matter so much? It's a precision vs. context trade-off:

🔍
Too small (< 100 tokens)

High precision — the right chunk is easy to find. But no context: a sentence about a gene mutation with no surrounding explanation is useless as an answer.

📚
Too large (> 1000 tokens)

Lots of context — but poor precision. "The Methods section" as one chunk means every methods query retrieves the same giant block, regardless of what subsection is relevant.

✨
Just right (300–512 tokens)

One coherent idea per chunk — a complete experiment description, a full statistical argument, a specific protocol. This is the sweet spot for academic RAG.

Why 512 specifically?

The cl100k_base tokenizer is used because it's the same one that OpenAI's embedding models use. The 512 hard maximum matches the historical context window of text-embedding-ada-002. If your embedding model supports more tokens, you can safely raise --max-tokens.

The fast path: when a section is short

The sliding window is only used for long sections. For a short section (like an Abstract), forcing a sliding window would create tiny overlapping fragments. The fast path prevents this:

chunker.py — _make_chunks(), fast path
    prefix = f"[Section: {section}] "
    prefix_toks = count_tokens(prefix)

        # Fast path: entire buffer fits in
    # one chunk — emit without overlap.
    full_body = " ".join(sentences)
    if (prefix_toks
            + count_tokens(full_body)
            <= max_tokens):
        return [_make_one(
            sentences,
            page_nums,
            chunk_index_start)]
Plain English
  1. prefix = f"[Section: {section}] " — Every chunk starts with its section name. This string counts against the token budget.
  2. prefix_toks — Count how many tokens the prefix uses, so we can subtract them from the available space for body text.
  3. full_body — Join all sentences in this section into one long string.
  4. if prefix_toks + count_tokens(full_body) ≤ max_tokens — "Does the entire section fit in one chunk?" If a 3-sentence Abstract uses only 180 tokens, why split it at all?
  5. return [_make_one(...)] — Return a list with exactly one chunk containing everything. No overlap, no sliding window. Done.
The tail-emission optimization

A second optimization handles the end of long sections. Without it, the sliding window generates a "shrinking spiral" of tiny tail chunks (e.g., 47 → 44 → 43 → 34 → 17 → 10 → 9 tokens). The fix: before starting a new window, check if all remaining sentences fit in one chunk. If yes, emit them as a final chunk and stop.

What a finished chunk looks like, how to interpret quality flags, and where this pipeline fits in the wider RAG world.

Anatomy of a chunk

Each line of chunks.jsonl is a JSON object with 8 fields. Here's a real example:

{
  "chunk_id":    "a3f8c2d1e5b9...",
  "text":        "[Section: Results] The kinase
                   activity was measured at 37°C...",
  "source_file": "1471-2199-11-10.pdf",
  "page_start":  4,
  "page_end":    5,
  "section":     "Results",
  "chunk_type":  "text",
  "chunk_index": 12,
  "token_count": 378
}
chunker.py — chunk_id & prefix generation
    prefix = f"[Section: {section}] "
    prefix_toks = count_tokens(prefix)
    ...
    return Chunk(
      chunk_id=hashlib.md5(
        (source_file + full)
        .encode("utf-8")
      ).hexdigest(),
      text=full,
      ...
    )
Plain English
  1. prefix = f"[Section: {section}] " — Every chunk's text starts with its section label. This isn't decorative — embedding models encode this prefix as part of the chunk's meaning, making retrieval dramatically more precise for section-specific queries.
  2. prefix_toks — The prefix itself uses tokens, so we subtract it from the budget before filling body text.
  3. hashlib.md5(...).hexdigest() — Generate a unique fingerprint for this chunk. MD5 of (source_file + full_text) guarantees uniqueness even if two different PDFs happen to contain the same paragraph.
  4. Why include source_file? — Without it, a paper and its preprint (identical text, different filenames) would produce the same chunk_id for every shared chunk. Including the filename makes IDs unique per-file.

Quality flags: the pipeline's health check

After processing, run python run.py --analyze chunks.jsonl to get an automatic quality report. These four flags cover the main failure modes:

FlagWhat it meansLikely cause
SHORT_CHUNKS Chunks < 50 tokens Very short section (e.g., a 2-sentence abstract); or sentence splitter fragmentation on special characters; or a heading that fired with no following body text
OVERSIZE_CHUNKS Chunks > 512 tokens A single sentence longer than max_tokens (e.g., a long DNA sequence inline in the methods, or a sentence with many parenthetical clauses)
DUPLICATE_IDS Same chunk_id appears twice The same PDF was processed twice (or a paper and its preprint were both included with the same filename)
SECTION_IMBALANCE >70% of all chunks in one section No detectable section boundaries in the paper — all content fell into the default "Introduction" bucket. Check if the paper uses non-standard heading styles.
